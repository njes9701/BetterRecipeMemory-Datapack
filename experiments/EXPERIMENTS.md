# 實驗報告:BetterRecipeMemory 正式環境實測

> 以下數據取自一台繁忙的正式 Minecraft 伺服器(Paper 26.2)。全篇比較兩台伺服器:
> **Server A** = 有裝 datapack(登錄表 127),**Server B** = 未裝的原版基準(登錄表 1688)。
> 原始直方圖在 [`data/`](data/);每個實驗都附完整算式,可自行重算。
> 原始檔裡有什麼、沒什麼,見 [`data/README.md`](data/README.md)。

## 摘要(30 秒版)

| 指標 | 舊(原版) | 新(裝 datapack) | 減量 |
|---|---|---|---|
| 每玩家 AdvancementProgress 物件 | **1,688** | **127** | **−92.5%** |
| 每玩家 CriterionProgress 物件 | 3,689 | 407 | −89.0% |
| 每登入人次的物化成本(實測,見 E6) | 0.5–1.0 MB | ~0.05 MB | **−90…95%** |
| 繁忙伺服器每日成就垃圾(~2,700 人次) | 1.3–2.6 GB/天 | ~0.13 GB/天 | ~−90% |
| 孤兒警告 log 洪水(裝配套外掛) | 150,660 行/晚 | 0 | −100% |

**因果證明的核心:比值精確整除。** 對照的兩台伺服器、三個時間點共六次取樣,
`成就物件 ÷ 玩家物件` 在新伺服器全部恆等於 **127.000**、舊伺服器恆等於 **1688.000**,
且取樣之間的增減完全 lockstep(E3)。這不是統計相關,是機制。

---

## 量測方法(讀懂數據前必看)

工具:`jcmd <pid> GC.class_histogram`(JDK 內建),對運行中 JVM 取「類別直方圖」——每個 class 的
實例數與 shallow bytes。

三個方法學重點:

1. **直方圖預設會先觸發一次 full GC**(除非加 `-all`),所以數到的都是**活物件**——這正是比值能
   精確整除的原因(死掉未回收的垃圾已被清走)。也因此它**不能**拿來量垃圾產生速率;垃圾速率是用
   「每人次物化成本 × 登入人次」推的(E5)。
2. 成本:每次一次 STW 暫停,實測 ~1–3 秒(像一次小 lag spike)。這**不是** heap dump(dump 會凍 30 秒以上,高壓時段禁用)。
3. `ServerPlayer` 實例數 ≈ 線上玩家 + 尚被強引用而未回收的離線玩家物件,**不等於**線上人數;
   但不影響比值論證——每個被物化的 ServerPlayer 都掛整套登錄表。

物件說明:
- `AdvancementProgress`:每玩家 × 每成就一個,記錄該成就的進度
- `CriterionProgress`:每個成就底下的判準各一個
- 兩者只是「掛在玩家身上的成就圖」的頂層;retained 全圖(HashMap、entry、key 字串)估約
  0.5–1 MB/玩家(E6 直接量到其中的結構本體)

## E1:跨主機普查——誰換了、誰還沒(log 掃描,零量測成本)

方法:掃每台伺服器開機 log 的 `Loaded N advancements`(該行只在開機印),不用碰 JVM。

結果:尚未換 datapack 的伺服器全部讀到精確的 **1688**;換好的讀到精確的 **127**(126 條原版留存
+ 本包 1 條隱形觸發成就);少數還在早期「純過濾」版的讀 **126**(無配方書回填)。重點:同樣兩個
載入數在**多台主機上獨立重現** —— 不是單一伺服器的個案。

**判讀:** `Loaded 127 advancements` 是最便宜的部署驗證指標——一行 log,不用碰 JVM。

---

## E2:跨伺服器 A/B——比值 127 vs 1688(heap 直方圖)

方法:同一晚對「已換」與「未換」各取直方圖,算 `AdvancementProgress ÷ ServerPlayer`。

樣本一:

| 伺服器 | 狀態 | ServerPlayer | AdvancementProgress | 比值 | CriterionProgress | 比值 |
|---|---|---|---|---|---|---|
| A | 新(datapack) | 20 | 2,540 | **2540÷20 = 127.000** | 8,140 | 407.0 |
| B | 基準 | 195 | 329,160 | **329160÷195 = 1688.000** | 719,355 | 3689.0 |

樣本二(原始檔 [`data/server-A_with-datapack_T0.txt.gz`](data/) / [`data/server-B_baseline_T0.txt.gz`](data/)):

| 伺服器 | 狀態 | ServerPlayer | AdvancementProgress | 比值 | CriterionProgress | 比值 |
|---|---|---|---|---|---|---|
| A | 新 | 44 | 5,588 | **5588÷44 = 127.000** | 17,908 | 407.0 |
| B | 基準 | 191 | 322,408 | **322408÷191 = 1688.000** | 704,599 | 3689.0 |

**判讀:** 四次取樣、兩台伺服器、人數各異,比值全部精確整除。這裡有兩本帳,別混:
- **資料檔帳**(掃 server jar):1,703 個成就檔,其中 1,572 個是 `minecraft:recipes/*` → 過濾後剩 131 檔
- **載入帳**(開機 log `Loaded N advancements`):1,688 → 127(126 條原版留存 + 本包隱形觸發成就);
  兩本帳的小差距(1,703 檔 vs 1,688 載入)是原版本來就有少數檔不進載入計數,與本包無關

每玩家物化數**恰等於載入數**(1688 與 127 都整除)→ 物化是無條件的、與玩家存檔內容無關;
這也是「清玩家檔救不了記憶體」的直接證據。

減量算式:
- AdvancementProgress:1688 → 127,**每玩家 −1,561 個(−92.48%)**
- CriterionProgress:3689 → 407,**每玩家 −3,282 個(−88.97%)**
- 兩類物件 shallow bytes(樣本二):B = (7,737,792+11,273,584)÷191 ≈ **97.3 KB/人**;
  A = (134,112+286,528)÷44 ≈ **9.3 KB/人**(−90.4%)。retained 全圖另含 map/entry/字串,E6 直接量。

## E3:時間序列 lockstep——增減與玩家數完全同步

方法:同兩台伺服器,間隔取樣相減(histo-diff)。

樣本一 → 樣本二(+26 分鐘):

| 伺服器 | ΔServerPlayer | ΔAdvancementProgress | 驗算 | ΔCriterionProgress | 驗算 |
|---|---|---|---|---|---|
| A(新) | 20→44(**+24**) | 2540→5588(**+3,048**) | 24 × 127 = 3,048 ✓ | +9,768 | 24 × 407 = 9,768 ✓ |
| B(基準) | 195→191(**−4**) | 329160→322408(**−6,752**) | 4 × 1688 = 6,752 ✓ | −14,756 | 4 × 3689 = 14,756 ✓ |

樣本二 → 樣本三(+31 分鐘;原始檔 `data/server-A_with-datapack_T1.txt.gz` / `data/server-B_baseline_T1.txt.gz`):

| 伺服器 | ΔServerPlayer | ΔAdvancementProgress | 驗算 | ΔCriterionProgress | 驗算 |
|---|---|---|---|---|---|
| A(新) | 44→71(**+27**) | 5588→9017(**+3,429**) | 27 × 127 = 3,429 ✓ | +10,989 | 27 × 407 = 10,989 ✓ |
| B(基準) | 191→140(**−51**) | 322408→236320(**−86,088**) | 51 × 1688 = 86,088 ✓ | −188,139 | 51 × 3689 = 188,139 ✓ |

(樣本三比值依然整除:A = 9017÷71 = 127.000;B = 236320÷140 = 1688.000 —— 六次取樣,六次整除)

**判讀:** Δ成就物件 = Δ玩家物件 × 每玩家物化條數(= 載入數 1688/127),一個物件不差。物化與釋放
完全跟著玩家生命週期走,徹底排除「隨機洩漏」假說;要縮小它,唯一的槓桿就是縮小登錄表(datapack
filter)或減少玩家 churn(不可控)。

## E4:log 噪音 A/B + 一個推翻預期的發現

背景:datapack 移除配方成就後,既有玩家存檔裡的舊條目變孤兒,join 載入時原版對每條印一行
`Ignored advancement ... doesn't exist anymore?`。原本的預期:印完後下次存檔即清,每玩家一生一次,
數天內自然消退。配套外掛 `BetterRecipeMemoryPlugin` 用 log4j root filter 吞掉這一種訊息。

### E4a:A/B 總量

同一晚,一台**未裝**配套外掛 vs 一台**已裝**,數 `latest.log`:

| 伺服器 | 配套外掛 | `Ignored advancement` 行數 | 涉及玩家檔數 | log 總行數 | 洪水佔比 |
|---|---|---|---|---|---|
| 未裝 | ✗ | **150,660** | 48 | 184,009 | **81.9%** |
| 已裝 | ✓ | **0** | — | 11,819 | 0% |

### E4b:分布鑑識——「一生一次」被實測推翻

平均 150,660 ÷ 48 ≈ 3,139 行/檔,**超過理論上限**(一個玩家檔最多 ~1,572 條孤兒)——有鬼。拆分布:

| 玩家檔(去識別) | 行數 | 佔比 |
|---|---|---|
| 檔案 #1(單一玩家) | **73,926** | **49.1%** |
| 檔案 #2 | 10,287 | 6.8% |
| 檔案 #3 | 7,242 | 4.8% |
| 其餘 45 檔 | 59,205 | 39.3% |

追查那個 73,926 行的檔案:
- 它的存檔有 **999 條孤兒**,73,926 ÷ 999 = **74 批,一批不差**
- 對照 log:該連線在 ~33 分鐘內 **join 了正好 74 次**(每 ~27 秒一次的重登迴圈,異常客戶端)
- → **每次 join 都整批重印**,不是一生一次

為什麼檔案沒被清?因為這台把 vanilla 成就存檔**關掉**了(`spigot.yml advancements.disable-saving: true`,
成就持久化交給外部跨服同步系統,vanilla json 只當 join 種子)——**檔案永遠不會被重寫,孤兒永遠在**,
每次 join 都重印一整批。

**判讀(重要):**
- vanilla 預設(成就存檔開啟):印一次 → 存檔清掉 → 數天自然消退,配套外掛可裝可不裝
- **關閉 vanilla 成就存檔**(如上):噪音永不消退,且被重登迴圈客戶端放大(單一連線 33 分鐘 7.4 萬行)→ 配套外掛實質必裝
- 額外收穫:這個分布順帶抓到一個每 ~27 秒重登一次的異常連線——log 噪音有時是免費的監控訊號

註:外掛上線前先以自測掛鉤驗證吞沒行為(`-Dbetterrecipememory.selftest=true` 開機丟一則誘餌訊息,
log 中不出現即通過)。

## E5:登入人次 × 每次成本 = 每日垃圾量(churn 帳)

方法:數 log 的 `logged in with entity id`(vanilla 必印,不受聊天訊息客製影響)。繁忙的一天,
Server A 約有 **2,700 人次登入**。

每日成就垃圾估算(每人次成本用 E6 實測區間):
- 舊:2,700 × 0.5–1.0 MB/人次 ≈ **1.3–2.6 GB/天** 的短命垃圾丟給 G1
- 新:2,700 × ~0.05 MB ≈ **0.13 GB/天**
- **省 ~1.2–2.5 GB/天**(繁忙伺服器)

**判讀:** 記憶體壓力不是「佔著不放的洩漏」,是「回收得掉但追不上的 churn 垃圾」——把離線玩家釘住
不放的那種記憶體洩漏是另一類獨立問題,治標不治本,縮登錄表才是對這條 churn 成本的根治。

## E6:每玩家實際 bytes 精算——「到底省幾 MB」

方法:利用 E3 的 lockstep 窗(Server B,T0→T1 恰好 −51 玩家),對兩份完整直方圖逐類相減。變化量
「整除 51 且對得上 1688/3689 倍數」的類別 = 成就鏈;通用類別(HashMap 等)以量級歸因。全程用 repo
內原始檔可重算。

**歸因驗證(Server B,−51 人):**

| 類別 | Δ實測 | 成就鏈預期 | 判定 |
|---|---|---|---|
| AdvancementProgress | −86,088(−2,066,112 B) | 51×1688 = 86,088 | ✓ 精確,24 B/個 |
| CriterionProgress | −188,139(−3,010,224 B) | 51×3689 = 188,139 | ✓ 精確,16 B/個 |
| java.util.HashMap | −87,252 | 51×1688(每成就一張 criteria map) | ✓ 誤差 1.3% |
| HashMap table 陣列 | −85,302(avg 124.6 B) | ≈51×1688 | ✓ 誤差 0.9% |
| HashMap$Node | −565,693 | 51×(3689+1688)=274,227(criteria 節點 + 進度表節點) | 成就鏈佔 48%,其餘為離線玩家的其他 map |
| java.time.Instant | −9,864(每人 193) | 已達成判準的時間戳 | ✓ 合理 |
| java.lang.String | **+51,468(不減反增)** | 若 key 每玩家複製應 −19 萬起跳 | **✓ 證明成就 key 字串是共享的,不算每人成本** |

**每玩家成就鏈實測(shallow 加總):**

| 組件 | 舊(1688/3689) | 新(127/407) |
|---|---|---|
| AdvancementProgress(24 B) | 40,512 B | 3,048 B |
| CriterionProgress(16 B) | 59,024 B | 6,512 B |
| criteria HashMap(48 B/張) | 81,024 B | 6,096 B |
| HashMap$Node(32 B,criteria + 進度表) | 172,064 B | 17,088 B |
| table 陣列(實測 avg ~125 B) | ~208,400 B | ~15,900 B |
| 進度表 table + Instant | ~12,800 B | ~1,800 B |
| **合計** | **~574 KB ≈ 0.55 MB/人** | **~50 KB ≈ 0.05 MB/人** |

**→ 實測每玩家省 ~0.5 MB(−91%)。**

與先前 ~1 MB/人 的 heap dump 估值差在哪:dump 量的是 retained,且含「從玩家 JSON 檔載入進度時新建
的 key 字串」(有進度的老玩家每人一份,不共享);直方圖法量到的是共享 key 以外的結構本體。
**誠實區間:每玩家 0.5 MB(結構實測下限)~ 1.0 MB(含檔案載入字串的 retained 上限)。**

**換算成「現在省了幾 MB」(以樣本三為準):**

| 情境 | 算式 | 結果 |
|---|---|---|
| Server A 常駐(71 人,已換) | 71 × 0.55 MB(若沒換)vs 71 × 0.05 MB | **40.7 → 3.5 MB,現省 ~37 MB** |
| Server B 常駐(140 人,未換) | 140 × 0.55 MB | 78 MB 掛在 heap 上;換了會變 7 MB,**可省 ~71 MB** |
| Server A 每日 churn 垃圾(2,700 人次) | 2,700 × (0.5–1.0 MB) → 2,700 × 0.05 MB | **1.3–2.6 GB/天 → 0.13 GB/天** |

**判讀:** 常駐節省(每台幾十 MB)只是配菜;主菜是 churn:多 GB 的 heap 每天少產生 1.2–2.5 GB
短命垃圾,才是真正紓解 G1 回收壓力與 mixed cycle 頻率的關鍵。

## 附:結論與再現步驟

1. 你的伺服器中招了嗎?一行診斷:
   ```bash
   jcmd <pid> GC.class_histogram | grep -E "AdvancementProgress|ServerPlayer\b"
   # AdvancementProgress ÷ ServerPlayer ≈ 1688(1.21.x)→ 中招
   ```
2. 裝 `dist/BetterRecipeMemory.zip` 到主世界 `datapacks/` → 重啟 → 開機 log 應見 `Loaded 127 advancements`
3. (可選)裝 `dist/BetterRecipeMemoryPlugin-26.2-1.jar` 消孤兒洪水
4. 重驗:比值應 ≈127

已知注意事項見 [README](../README.md):`minecraft:limited_crafting` 必為 false、pack.mcmeta 的
`min_format`/`max_format` 必填、勿用 `minecraft:reload` 熱載(會洗掉外掛自訂配方)。
